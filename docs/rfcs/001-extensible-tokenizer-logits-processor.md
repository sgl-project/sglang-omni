# RFC-001: Extensible Tokenizer and Logits Processor Architecture

**Status**: Draft
**Authors**: [TBD]
**Created**: 2026-02-20
**Target**: sglang-omni `engines/omni/runtime`

## Motivation

sglang-omni's current AR runtime hardcodes two assumptions:

1. **Tokenizer**: `create_ar_engine` accepts an opaque `tokenizer` object and only reads
   `eos_token_id` from it. All prompt construction, special-token handling, and
   vocabulary mapping happen outside the engine. This works for HuggingFace chat
   models but breaks for models with non-standard tokenizers (e.g., FishAudio-S1's
   tiktoken-based `FishTokenizer` with 4096 semantic tokens and interleaved
   multimodal prompt format).

2. **Logits processing**: `AROutputProcessor.process()` performs a single
   temperature-scaled sampling (`argmax` or `multinomial`). There is no support
   for top-p, repetition penalty, frequency penalty, or model-specific
   post-processing (e.g., DualAR's multi-codebook sampling). Adding a new
   sampling strategy requires replacing the entire `OutputProcessor`.

Both limitations become blockers when integrating new model families.
Rather than adding model-specific forks, we propose two composable abstractions
that plug into the existing Protocol-based architecture.

### Why FishAudio-S1 Needs All Three Sampling Parameters

The current `AROutputProcessor` only supports `temperature`. FishAudio-S1's
`logits_to_probs()` requires three parameters applied in a strict order:

1. **`repetition_penalty`** (default 1.1) — TTS models generate ~86 VQ tokens
   per second of audio. Without penalizing repeated tokens, the model easily
   falls into loops that produce audible "skipping" artifacts. The penalty
   scales already-seen token scores: positive scores are divided by the penalty,
   negative scores are multiplied.

2. **`top_p`** (default 0.8) — Each codebook has 1024 discrete entries
   representing acoustic features. Sampling from the full distribution
   occasionally selects low-probability tokens that produce noise, pops, or
   sudden pitch shifts. Nucleus sampling truncates the long tail, keeping only
   the token set whose cumulative probability reaches `top_p`.

3. **`temperature`** (default 0.8) — Controls the trade-off between
   expressiveness and stability. Too low (→0) produces flat, robotic speech;
   too high (→1+) introduces instability and artifacts. The default 0.8
   balances naturalness with reliability.

The execution order matters: `repetition_penalty → top_p → temperature →
softmax → sample`. Reordering (e.g., applying temperature before top-p) shifts
the nucleus boundary and degrades quality.

### Why DualAR Generates `num_codebooks+1` Tokens Per Step

FishAudio-S1 uses a **Residual Vector Quantization (RVQ)** audio codec (DAC)
that compresses audio into multiple codebook layers:

| Layer | Encodes | Analogy |
|-------|---------|---------|
| Codebook 0 (semantic) | Phonemes, pitch, prosody, timing | Pencil sketch |
| Codebook 1–3 | Timbre, formants, breathiness | Color and shading |

Each layer encodes the **quantization residual** of the previous layer.
Codebook 0 roughly approximates the original audio feature; codebook 1
approximates the error left by codebook 0; and so on.

The **semantic token** (codebook 0) is generated by the **slow transformer**
(the main LLM backbone). This is the core text-to-speech decision: what
phoneme, how long, what pitch contour. It requires full context attention
over the entire prompt.

The remaining **codebook tokens** (1 through `num_codebooks-1`) are generated
by the **fast transformer** (a small 4-layer network), conditioned on the
semantic token's hidden state. These add acoustic detail — without them the
decoded audio sounds muffled and metallic.

The two transformers are separate because:

- **Codebooks have sequential dependencies**: codebook 1's value depends on
  codebook 0's choice, codebook 2 depends on 0 and 1, etc. Parallel prediction
  significantly degrades quality.
- **Computational asymmetry**: the slow transformer needs full context (many
  layers, large KV cache). The fast transformer only needs the current step's
  hidden state (4 layers, KV cache reset every step). This keeps total compute
  close to a single LLM forward pass.

Each decode step therefore produces `[num_codebooks+1, 1]`: 1 semantic token
from the slow transformer + `num_codebooks` acoustic tokens from the fast
transformer.

## Design Goals

1. **Backward compatible** – existing `create_ar_engine` / `create_encoder_engine`
   continue to work without changes.
2. **Composable** – logits processors can be stacked (temperature → top-p →
   repetition penalty → model-specific).
3. **Model-agnostic** – the same interfaces work for standard LLMs, DualAR TTS,
   DiT, or future model types.
4. **Zero overhead for simple cases** – when no logits processors are configured,
   the hot path remains a single `argmax`.

## Detailed Design

### 1. TokenizerAdapter Protocol

```python
# sglang_omni/engines/omni/runtime/tokenizer.py

class TokenizerAdapter(Protocol):
    """Minimal contract between the engine and any tokenizer."""

    @property
    def vocab_size(self) -> int: ...

    @property
    def eos_token_ids(self) -> list[int]:
        """One or more token IDs that signal generation should stop."""
        ...

    def encode(self, text: str) -> list[int]: ...

    def decode(self, token_ids: list[int]) -> str: ...
```

#### Provided Implementations

| Class | Wraps | Notes |
|-------|-------|-------|
| `HFTokenizerAdapter` | Any HuggingFace `PreTrainedTokenizer` | Default, reads `eos_token_id` / `chat_template` |
| `FishTokenizerAdapter` | `FishTokenizer` | Maps `<\|im_end\|>` to eos, exposes `semantic_begin_id` / `semantic_end_id`, builds `ContentSequence` prompts |

#### Prompt Builder (optional extension)

For models whose prompt format is not a simple chat template, the adapter can
expose a `build_prompt` method:

```python
class PromptBuilder(Protocol):
    def build_prompt(
        self,
        text: str,
        references: list[Reference] | None = None,
        **kwargs,
    ) -> torch.Tensor:
        """Build model-ready input tensor(s) from user request.

        Returns shape varies by model:
        - Standard LLM:  [seq_len]
        - DualAR:        [num_codebooks+1, seq_len]
        """
        ...
```

This keeps the engine itself free of model-specific prompt logic while
making the prompt construction discoverable and testable.

**Why the existing hardcoded approach is insufficient**: The current
`ARInputPreparer` assumes prompts are flat 1-D token ID sequences obtained
by calling `tokenizer.encode()`. FishAudio-S1's prompt is a multi-row tensor
of shape `[num_codebooks+1, seq_len]` where row 0 holds text/semantic tokens
and rows 1–N hold VQ codes. This interleaving is constructed through
`ContentSequence`, which alternates `TextPart` and `VQPart` segments with
speaker and modality markers. There is no `chat_template` — the "prompt format"
is an imperative sequence of `seq.append(parts, speaker=...)` calls. Moving
this logic into a `PromptBuilder` extension means the engine never needs to
know about `ContentSequence`, `VQPart`, or speaker tags; it only calls
`build_prompt(text, references)` and receives a ready-to-use tensor.

### 2. LogitsProcessor Protocol and Pipeline

```python
# sglang_omni/engines/omni/runtime/logits_processor.py

class LogitsProcessor(Protocol):
    def __call__(
        self,
        logits: torch.Tensor,
        context: SamplingContext,
    ) -> torch.Tensor:
        """Transform logits in-place or return new logits.

        Args:
            logits: [batch, vocab_size] or [batch, codebook_size] raw logits
            context: per-request sampling state (previous tokens, params, etc.)
        """
        ...


@dataclass
class SamplingContext:
    """Per-request state available to logits processors."""
    request_id: str
    previous_tokens: torch.Tensor | None = None
    temperature: float = 1.0
    top_p: float = 1.0
    top_k: int = -1
    repetition_penalty: float = 1.0
    step: int = 0
    metadata: dict[str, Any] = field(default_factory=dict)


class LogitsProcessorPipeline:
    """Ordered chain of LogitsProcessor instances."""

    def __init__(self, processors: list[LogitsProcessor] | None = None):
        self._processors = processors or []

    def add(self, processor: LogitsProcessor) -> "LogitsProcessorPipeline":
        self._processors.append(processor)
        return self

    def __call__(
        self, logits: torch.Tensor, context: SamplingContext
    ) -> torch.Tensor:
        for proc in self._processors:
            logits = proc(logits, context)
        return logits
```

#### Built-in Processors

| Processor | Behavior |
|-----------|----------|
| `TemperatureProcessor` | `logits / max(temperature, 1e-5)` |
| `TopPProcessor` | Nucleus sampling mask |
| `TopKProcessor` | Top-k truncation |
| `RepetitionPenaltyProcessor` | Scale repeated token logits |
| `FrequencyPenaltyProcessor` | Penalize by token frequency |

#### Sampler

After the pipeline, a `Sampler` converts processed logits to token IDs:

```python
class Sampler(Protocol):
    def sample(
        self,
        logits: torch.Tensor,
        context: SamplingContext,
    ) -> SamplerOutput: ...


@dataclass
class SamplerOutput:
    token_ids: torch.Tensor      # [batch] or [batch, num_codebooks+1]
    logprobs: torch.Tensor | None = None
```

Standard `ArgmaxSampler` and `MultinomialSampler` are provided.
DualAR adds `DualARSampler` that first samples the semantic token from
`token_logits`, then autoregressively samples codebook tokens from the fast
transformer.

**Why separate the Sampler from LogitsProcessors**: Logits processors are
purely functional transforms on the logits tensor — they don't produce token
IDs. The `Sampler` is the boundary between "probability distribution
manipulation" and "discrete decision". This separation matters because:

- The same pipeline (temperature + top-p + repetition penalty) works with
  different samplers (`ArgmaxSampler` for greedy, `MultinomialSampler` for
  stochastic, `MultinomialNoSyncSampler` for FishAudio's non-synchronized
  multinomial using the Gumbel-max trick).
- The sampler can optionally return `logprobs`, decoupling probability
  computation from token selection.
- DualAR's two-stage decode needs two calls to `Sampler.sample()` (once for
  semantic, once per codebook) with different logits tensors but similar
  pipeline processing.

**Why LogitsProcessorPipeline is a chain**: The ordering contract is
`repetition_penalty → top_p → temperature` (see Motivation). By making
the pipeline an explicit ordered list, we:

- Make the execution order visible and debuggable (just print the list).
- Allow per-model customization by appending model-specific processors
  (e.g., a hypothetical `SemanticRangeMask` that zeroes out non-semantic
  logits during slow transformer sampling).
- Enable easy A/B testing by swapping one processor in the chain.

### 3. Integration with Existing Architecture

The existing `OutputProcessor` protocol stays unchanged. We refactor
`AROutputProcessor` to delegate to the new pipeline:

```
┌─────────────────────────────────────────────────────┐
│  ModelRunner.execute()                              │
│    ┌──────────────┐    ┌────────────────────────┐   │
│    │ InputPreparer │───▶│ model(**inputs)         │   │
│    └──────────────┘    └───────────┬────────────┘   │
│                                    │                │
│                          model_output.logits        │
│                                    │                │
│    ┌───────────────────────────────▼──────────────┐ │
│    │  OutputProcessor.process()                    │ │
│    │    │                                          │ │
│    │    ├─ build SamplingContext per request        │ │
│    │    ├─ LogitsProcessorPipeline(logits, ctx)    │ │
│    │    ├─ Sampler.sample(logits, ctx)             │ │
│    │    └─ return RequestOutput(token_ids, ...)     │ │
│    └───────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

#### Factory Changes

```python
def create_ar_engine(
    model,
    tokenizer=None,
    *,
    logits_processors: list[LogitsProcessor] | None = None,
    sampler: Sampler | None = None,
    **kwargs,
) -> OmniEngine:
    adapter = wrap_tokenizer(tokenizer)  # auto-detect HF vs Fish
    pipeline = LogitsProcessorPipeline(logits_processors)
    sampler = sampler or MultinomialSampler()

    output_processor = AROutputProcessor(
        logits_pipeline=pipeline,
        sampler=sampler,
    )
    ...
```

### 4. DualAR-Specific Extensions

For FishAudio-S1's DualARTransformer, the model outputs
`TransformerForwardResult(token_logits, codebook_logits)` instead of a single
logits tensor. The extension adds:

```python
@dataclass
class DualARSamplerOutput(SamplerOutput):
    """Extended output with per-codebook tokens."""
    semantic_token: int = 0
    codebook_tokens: torch.Tensor | None = None  # [num_codebooks]


class DualAROutputProcessor(OutputProcessor):
    """Handles the two-stage sampling of DualARTransformer.

    1. Sample semantic token from slow transformer's token_logits
    2. Autoregressively sample codebook tokens from fast transformer
    """
    def __init__(
        self,
        slow_logits_pipeline: LogitsProcessorPipeline,
        fast_logits_pipeline: LogitsProcessorPipeline,
        sampler: Sampler,
        num_codebooks: int = 4,
    ): ...
```

This plugs into the existing `ModelRunner` / `OmniEngine` without changing
any core code — only `create_dual_ar_engine()` is new.

**How DualAR integrates without changing core code**: The `OmniEngine` loop
is model-agnostic — it calls `Scheduler → ModelRunner → OutputProcessor` in
a loop. Each component is a Protocol. DualAR integration only provides new
*implementations* of these protocols:

| Core Protocol | DualAR Implementation | Key Difference |
|---------------|----------------------|----------------|
| `BatchPlanner` | `DualARBatchPlanner` | Tracks multi-row token budget (each token = `num_codebooks+1` rows) |
| `InputPreparer` | `DualARInputPreparer` | Builds `[num_codebooks+1, batch_seq_len]` input tensors instead of `[batch, seq_len]` |
| `OutputProcessor` | `DualAROutputProcessor` | Two-stage sampling: slow pipeline → semantic, then fast pipeline × N → codebooks |
| `IterationController` | `DualARIterationController` | Stops on `<\|im_end\|>` semantic token instead of a simple EOS check |
| `CacheManager` | `DualARRadixCache` | Keys on row-0 (semantic tokens) for prefix sharing, stores all rows |

The `OmniEngine`, `Scheduler`, and `ModelRunner` classes remain untouched.
New models follow the same pattern: implement the protocols, register a
factory function.

### 5. Migration Path

| Phase | Change | Breaking? | Detail |
|-------|--------|-----------|--------|
| Phase 1 | Add `TokenizerAdapter`, `LogitsProcessor`, `Sampler` protocols + built-in impls | No | Pure additions — new files under `runtime/`. No existing code is modified. Existing tests continue to pass. |
| Phase 2 | Refactor `AROutputProcessor` to use pipeline internally | No (same external behavior) | `AROutputProcessor.__init__` gains optional `logits_pipeline` and `sampler` kwargs with backward-compatible defaults (`pipeline=[]`, `sampler=ArgmaxSampler`). When no processors are configured, the hot path is identical to today: a single `argmax` call. |
| Phase 3 | Add `FishTokenizerAdapter` + `DualAROutputProcessor` | No (additive) | New classes only. `create_dual_ar_engine()` is a new factory entry point; `create_ar_engine()` is unchanged. Users who don't use DualAR never see these classes. |
| Phase 4 | Deprecate direct `temperature` field on `ARRequestData` in favor of `SamplingContext` | Minor | `ARRequestData.temperature` still works but emits a deprecation warning. The field is forwarded into `SamplingContext.temperature` internally. Full removal is deferred to a future major version. |

## Open Questions

### Q1: Should `SamplingContext` carry a reference to the `TokenizerAdapter`?

Some `LogitsProcessor` implementations need vocab-level information. For
example, a `SemanticRangeMaskProcessor` for DualAR needs to know
`semantic_begin_id` / `semantic_end_id` to mask non-semantic logits during
slow transformer sampling.

| Option | Approach | Pros | Cons |
|--------|----------|------|------|
| **A (recommended)** | Inject via processor constructor | Low coupling; `SamplingContext` stays lightweight and serializable; processor is self-contained after construction | Processor needs to be constructed with tokenizer info upfront; slightly more boilerplate |
| **B** | Add `tokenizer: TokenizerAdapter` field to `SamplingContext` | Any processor can access any vocab info at runtime; zero constructor boilerplate | `SamplingContext` becomes heavier; harder to serialize/log; creates implicit dependency between processor and tokenizer |
| **C** | Use `metadata: dict` on `SamplingContext` | Flexible; no protocol changes needed | Untyped; easy to misuse; no IDE auto-complete or type checking |

**Leaning**: Option A. Vocab info is static per model — it doesn't change
between requests. Passing it at construction time is natural, and keeps
`SamplingContext` purely about per-request sampling state.

### Q2: Should `LogitsProcessorPipeline` support async processors?

Certain advanced techniques require async execution during logits processing:
- **Classifier-Free Guidance (CFG)**: needs a second forward pass (unconditional)
  and blends two logits tensors.
- **External Reward Models**: queries a separate model/service to re-score
  candidate tokens.

| Option | Approach | Pros | Cons |
|--------|----------|------|------|
| **A (recommended)** | Keep pipeline sync; model CFG as a separate `OutputProcessor` wrapper | Hot path stays simple; no `async` contamination in the common case; CFG is architecturally different from per-token logits transforms | CFG can't be "just another processor in the chain" |
| **B** | Make pipeline `async def __call__` | Uniform interface; any processor can do I/O | All processors must be `async` even when they don't need it; performance overhead from coroutine creation on every token; complicates CUDA graph capture |
| **C** | Support both sync and async via overloading | Maximum flexibility | Complex implementation; two code paths to maintain; hard to reason about execution order |

**Leaning**: Option A. Logits processors should be fast, synchronous, in-place
tensor operations. CFG and reward models are fundamentally different — they
need extra forward passes, not just logits transforms. They belong at the
`OutputProcessor` or `ModelRunner` level.

### Q3: DualAR fast transformer decoding — N pipeline calls or one atomic step?

Each decode step needs `num_codebooks` sequential codebook tokens from the fast
transformer. Two modeling choices:

| Option | Approach | Pros | Cons |
|--------|----------|------|------|
| **A** | N calls to `fast_pipeline → sampler.sample()` | Per-codebook temperature/top_p tuning; reuses the same pipeline abstraction; easy to add per-codebook processors | 4× pipeline overhead per decode step; over-abstraction for what is essentially an inner loop |
| **B (recommended)** | Single atomic step inside `DualAROutputProcessor` | Minimal overhead; the fast transformer loop can be fused/optimized (e.g., single CUDA graph replay); matches FishAudio's reference implementation structure | Less flexible if different codebooks need radically different sampling strategies |
| **C** | Hybrid — single atomic step but accept a `list[LogitsProcessorPipeline]` (one per codebook) | Per-codebook customization without N separate calls; clear intent | More complex API; unclear if per-codebook customization is ever needed in practice |

**Leaning**: Option B. FishAudio's reference code uses the same
`temperature` / `top_p` for all codebooks. Per-codebook tuning is speculative.
Start simple, refactor to C if a concrete use case emerges.

## Appendix: SGLang `CustomLogitProcessor` Design Analysis

SGLang main provides a `CustomLogitProcessor` mechanism for user-defined logits
manipulation. We analyzed the full call chain to inform our design.

### Architecture

```
Client                              Server
──────                              ──────
CustomLogitProcessor.to_str()  ──HTTP──▶  CustomLogitProcessor.from_str()
  (dill serialize to hex)                   (dill deserialize, lru_cache)
                                                     │
                                    SamplingBatchInfo.from_schedule_batch()
                                      ├─ merge same-type processors
                                      └─ build batch mask tensor per processor
                                                     │
                                    Sampler.forward()
                                      ├─ _preprocess_logits()
                                      │    └─ apply_custom_logit_processor()   ← runs FIRST
                                      ├─ NaN detection
                                      ├─ temperature scaling
                                      ├─ softmax → probs
                                      ├─ top_k + top_p + min_p filtering       ← hardcoded
                                      └─ multinomial sampling
```

Key implementation details from `sglang/srt/sampling/`:

1. **Serialization for remote execution**: `CustomLogitProcessor` uses `dill` to
   serialize the entire class and transmit it as a hex string over HTTP. Server
   caches deserialized instances via `@lru_cache`. This is necessary because
   SGLang is a client-server architecture where the user defines processors on
   the client side.

2. **Batch-aware processing**: The processor receives `logits[batch, vocab]` for
   the entire batch. `SamplingBatchInfo` maintains a `Dict[int, Tuple[processor,
   mask_tensor]]` keyed by processor hash, where `mask_tensor` is a bool tensor
   indicating which requests in the batch use that processor. On invocation,
   only the masked subset is passed to the processor:

   ```python
   # sglang/srt/layers/sampler.py::apply_custom_logit_processor()
   for _, (processor, batch_mask) in sampling_batch_info.custom_logit_processor.items():
       indices = batch_mask.nonzero(as_tuple=True)[0]
       custom_param_list = [{**params[i], "__req__": reqs[i]} for i in indices]
       logits[indices] = processor(logits[indices], custom_param_list)
   ```

3. **Request state injection**: Each processor receives a `custom_param_list`
   where each dict is augmented with `"__req__"` — the full `Req` object
   containing `origin_input_ids`, `output_ids`, etc. This is how
   `ThinkingBudgetLogitProcessor` counts thinking tokens and
   `DeepseekOCRNoRepeatNGramLogitProcessor` accesses the full generation history.

4. **`custom_params` passthrough**: `SamplingParams.custom_params: Dict[str, Any]`
   is user-specified per request and flows all the way to the processor. This
   allows per-request configuration (e.g., `{"token_id": 5}` for deterministic
   sampling, `{"thinking_budget": 100}` for thinking length control).

5. **Built-in sampling is NOT modular**: `temperature`, `top_p`, `top_k`,
   `min_p` are flat fields on `SamplingBatchInfo`, applied by hardcoded logic
   in `Sampler._sample_from_probs()`. `repetition_penalty`, `frequency_penalty`,
   `presence_penalty` are handled by a separate `penaltylib.BatchedPenalizerOrchestrator`.
   None of these are modeled as `CustomLogitProcessor` instances.

### What We Adopt — Detailed Plan

#### Adopt 1: `custom_params` passthrough (Phase 1)

**SGLang pattern**: `SamplingParams.custom_params: Dict[str, Any]` flows from
user request all the way into the processor's `custom_param_list`.

**Our adoption**: `SamplingContext.metadata: dict[str, Any]` serves the same
role. When constructing `SamplingContext` in `OutputProcessor.process()`, the
engine copies request-level custom params into `metadata`.

**Concrete benefit for FishAudio-S1**: Voice cloning requests need per-request
parameters like `speaker_id`, `speed_ratio`, `emotion_weight`. Without
passthrough, every new parameter requires adding a typed field to
`DualARRequestData` and modifying the `OutputProcessor` — a multi-file change.
With `metadata`, the user sets `custom_params={"speaker_id": 3, "speed_ratio":
1.2}` on the request, and a `SpeakerConditioningProcessor` reads them directly
from `context.metadata["speaker_id"]`.

**Concrete benefit for future models**: A `ThinkingBudgetProcessor` (like
SGLang's) can read `context.metadata["thinking_budget"]` to control reasoning
length — no engine changes needed, just register the processor.

**Implementation cost**: Minimal — add one line in `OutputProcessor.process()`
to forward `request.custom_params` into `SamplingContext.metadata`.

#### Adopt 2: Request state injection (Phase 1)

**SGLang pattern**: `__req__` key injected into `custom_param_list`, giving
processors access to `origin_input_ids`, `output_ids`, etc.

**Our adoption**: Add typed fields to `SamplingContext`:

```python
@dataclass
class SamplingContext:
    # ... existing fields ...
    previous_tokens: torch.Tensor | None = None  # all generated tokens so far
    input_token_ids: torch.Tensor | None = None   # original input tokens
    step: int = 0                                  # current decode step
    metadata: dict[str, Any] = field(default_factory=dict)
```

**Why typed fields instead of SGLang's `__req__` hack**: SGLang injects the
entire `Req` object as an untyped dict entry. This works but has no IDE
auto-complete, no type checking, and couples the processor to SGLang's internal
`Req` class. Our typed `previous_tokens` / `input_token_ids` / `step` fields
provide the same information with full type safety.

**Concrete benefit for FishAudio-S1**: `RepetitionPenaltyProcessor` needs
`previous_tokens` to know which tokens to penalize. Without request state,
the processor can't see what's been generated — it would have to be stateful
and track tokens externally, which breaks the functional pipeline model.

**Concrete benefit for future models**: An `NGramBlockProcessor` needs the
full generation history to detect n-gram repetitions (exactly what SGLang's
`DeepseekOCRNoRepeatNGramLogitProcessor` does via `__req__`). With
`context.previous_tokens`, this is straightforward.

**Implementation cost**: Minimal — `SamplingContext` already has
`previous_tokens` and `step`. Add `input_token_ids` (one field).

#### Adopt 3: Batch-aware interface (Phase 2, performance optimization)

**SGLang pattern**: Processor receives `logits[batch, vocab]` for the entire
batch, with a mask tensor selecting relevant requests. One kernel call for
the whole batch.

**Our current design**: Per-request `SamplingContext` — the pipeline is called
once per request with `logits[1, vocab]`.

**Why defer to Phase 2**: Per-request is simpler to implement and debug. For
FishAudio-S1's initial integration, batch sizes will be small (TTS latency
is dominated by audio decoding, not sampling). The abstraction cost of
batch-aware processors is not justified yet.

**When to upgrade**: When profiling shows that per-request pipeline overhead
is significant (likely at batch_size > 32 with multiple processors). The
upgrade path is backward-compatible:

```python
class LogitsProcessor(Protocol):
    def __call__(
        self,
        logits: torch.Tensor,           # [batch, vocab] in batch mode
        context: SamplingContext | list[SamplingContext],
    ) -> torch.Tensor: ...
```

Existing single-request processors continue to work (the pipeline wraps them
in a loop). New batch-optimized processors can process the full tensor at once.

**Concrete benefit**: At batch_size=64 with 3 processors, batch mode reduces
kernel launches from 64×3=192 to 3. For processors that are purely element-wise
(temperature, top_p), this is a significant speedup.

**Implementation cost**: Moderate — requires `LogitsProcessorPipeline` to
detect and dispatch between per-request and batch modes.

#### Summary

| Adoption | Phase | Cost | Primary Benefit |
|----------|-------|------|----------------|
| `custom_params` passthrough | Phase 1 | 1 line | Per-request flexibility without schema changes |
| Request state injection (typed) | Phase 1 | 1 field | Processors can access generation history for penalties/blocking |
| Batch-aware interface | Phase 2 | Moderate | 10-60× fewer kernel launches at large batch sizes |

### Where We Diverge

| Aspect | SGLang | Our Design | Rationale |
|--------|--------|-----------|-----------|
| Built-in sampling | Hardcoded in `Sampler` class | Also modeled as processors in the pipeline | FishAudio requires `repetition_penalty → top_p → temperature` in a specific order; making them processors allows per-model reordering without changing engine code |
| Serialization | `dill` over HTTP | In-process, no serialization | sglang-omni processors are registered at engine construction time, not sent over the network |
| Composition | Single processor, all logic internal | Chain of processors | Enables mix-and-match: e.g., reuse `RepetitionPenaltyProcessor` across models, add model-specific `SemanticRangeMask` only for DualAR |
| Processor scope | Hook that runs before built-in sampling | Unified pipeline that IS the sampling logic | Avoids the split where some transforms are "custom" and others are "built-in" — everything is a processor, ordered explicitly |

### Conclusion

SGLang's `CustomLogitProcessor` is a pragmatic extension point for a
client-server LLM serving engine. Our design takes the useful patterns
(`custom_params` passthrough, request state access, batch-aware potential)
while going further: unifying built-in and custom sampling into one
composable pipeline. This is justified because sglang-omni needs to support
model families (DualAR TTS) whose sampling logic is fundamentally different
from standard LLMs, not just slightly customized.
